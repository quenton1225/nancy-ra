# 80 组会 PPT 改版构思

**日期**: 2026-04-29  
**对应旧版**: `80_next_meeting_group_presentation_20260429.html`  
**目的**: 将当前“给自己/项目内部看的 PPT”改成“给导师组做判断和提建议的 PPT”。

## 1. 总体问题

当前 80 版最大的问题不是内容错误，而是受众错位。

它现在像是在向项目内部说明“我们做了什么”，但导师组更需要看到：

1. 这个方向的研究空白在哪里；
2. 我们的工作和前人有什么关系；
3. 当前 demo 证明了什么、没有证明什么；
4. 下一步具体实验任务应该怎么设计；
5. 需要导师在哪些节点给判断。

因此新版 PPT 的核心姿态应从：

> 我们已经做了这些工作。

改为：

> 我们发现了一个可做但尚未充分验证的研究空白；我们已经做了 feasibility proof；现在需要导师确认下一步任务设计和数据边界。

## 2. 方法论缺口页必须提前

### 2.1 当前证据能说什么

| 证据 | 能支持什么 | 不能支持什么 |
| --- | --- | --- |
| C007 / WebGazer neurodivergence learning task | 普通浏览器 WebGazer 可以在学习任务中采集 gaze，并用于学习状态/注意/理解相关建模；原文也强调不能用于诊断或单独标记学生。 | 不能直接证明 SEN/SEND 大学生支持需求可以被准确推断。 |
| A5 | 有人尝试把多模态特征用于 special needs 个性化教育模型。 | 对象是 5-18 岁低龄学习者，且数据来自公开/派生数据，不是成年人或大学在校生的真实平台任务。 |
| A6 | sensor/process data 可以作为 individualized/differentiated education 的概念依据。 | 是 opinion article，不是实证证据。 |
| A7 | 大学生写作任务中，过程数据和 SRL 序列能揭示最终成绩/动机量表看不到的差异。 | 不是 SEN/SEND 特定群体，也不是 webcam eye tracking。 |
| A1-A4 | SEN/SEND / learning difficulties 在高等教育中确有支持需求、系统障碍和技术可信度问题。 | 没有直接验证 webcam gaze + interaction logs 的预测价值。 |

### 2.2 新版必须单独有一页

建议新增一页标题：

> **研究空白：成人/大学 SEN/SEND 场景下，普通笔记本过程数据是否能理解支持需求？**

这一页的主结论：

> 前人已经分别做过三件事：学习任务中的 WebGazer、special needs 的多模态/自适应模型、大学生学习过程数据分析。但三者尚未在“大学 SEN/SEND 支持需求”这个场景中合在一起充分验证。因此我们的贡献不是声称已经解决问题，而是提出一个可验证的 feasibility-to-pilot 路线。

建议该页结构：

| 前人已做 | 我们看到的缺口 | 我们下一步 |
| --- | --- | --- |
| WebGazer 学习任务 | 多用于 neurodivergence/attention/comprehension，不直接等于 SEN/SEND 支持需求 | 复现普通笔记本采集链路 |
| 多模态 special needs 模型 | 低龄、派生数据、非高等教育真实任务 | 只借鉴多模态思路，不夸大证据 |
| 大学生 process data / SRL | 有大学生过程分析，但不针对 SEN/SEND，也不一定用 gaze | 把 gaze + pointer + timing 作为任务过程信号 |

## 3. “页面上到底做什么活动”必须变成导师决策页

旧版 80 默认了 reading task，但没有向导师解释为什么选阅读、还可以选什么、页面应该长什么样。这会让导师觉得方案还停在工程 demo，不是研究设计。

新版应增加一页：

> **下一版任务活动设计：我们应该让学生在页面上做什么？**

建议给导师三个候选任务，而不是只给一个结论。

| 方案 | 页面活动 | 优点 | 风险/不足 | 适合作为下一步吗 |
| --- | --- | --- | --- | --- |
| A. 阅读理解任务 | 阅读短文本，回答理解题，穿插 TUT/困惑 probe | 最接近 C007，眼动解释最直接，适合 AOI 和 scroll | 可能偏认知心理，不够贴近真实课程平台 | **推荐第一版** |
| B. 学习材料 + 支持选择 | 阅读/观看学习材料，遇到困难时选择提示、摘要、延长时间等支持 | 更贴近 functional support needs，可直接观察支持偏好 | 页面和任务设计更复杂，需要定义支持选项 | 推荐第二版 |
| C. 轻量小游戏/互动任务 | 简单规则学习、记忆或注意任务 | 可控性强，数据干净，适合实验 | 与高等教育学习支持距离较远，导师可能质疑生态效度 | 暂不推荐作为主线 |

建议给出的判断：

> 第一版继续用“短文本阅读理解 + thought probe + 支持偏好询问”。它最容易和 C007 对齐，也最容易解释 gaze/pointer/scroll。会后第二版再加入“可选支持工具”，例如摘要、提示、延长阅读时间、术语解释等。

### 3.1 页面风格也要给导师看

新版还应有一小块说明页面风格：

- 不做游戏化、花哨或压力测试式页面；
- 做成安静、可访问、低认知负担的学习页面；
- 字体足够大，行距清楚；
- 文本区、问题区、选项区、按钮区布局稳定，方便 AOI 定义；
- 每一屏只做一件事，避免 UI 变化干扰 gaze 解释；
- 明确告诉受试者数据用途是学习支持研究，不是能力评判。

## 4. Part C 需要从“一页概览”拆成多页字段说明

用户指出得对：当前 Part C 的三张小卡片对技术人员够用，但对管理学背景导师不够友好。新版应把每一批字段单独成页，用“技术字段 + 中文含义 + 为什么收 + 风险控制”的格式解释。

### 4.0 Pre/Post 问卷需要单独一页

除了网页端过程数据，第一阶段还需要 pre/post questionnaire。原因是 gaze、pointer、scroll 和 response latency 本身不是状态标签；问卷可以提供解释参照，让我们知道过程信号是否真的对应注意、困惑、压力、支持偏好或技术信任。

建议新增一页：

> **Pre/Post 问卷：给过程数据提供解释标签**

| 阶段 | 领域 | 希望收集的信息 | 用途 |
| --- | --- | --- | --- |
| Pre-task | 学习背景 | 专业阶段、材料熟悉度、相关先验知识 | 解释阅读速度和理解差异 |
| Pre-task | 当前状态 | 疲劳、压力、注意力自评、睡眠或精神状态 | 作为状态分类或聚类的参照标签 |
| Pre-task | 技术与隐私 | 对 webcam tracking、数据使用和系统判断的信任度 | 呼应 A4 的技术可信度问题 |
| Pre-task | 支持经历 | 是否常用摘要、延时、术语解释、分步提示等支持 | 建立 support preference baseline |
| Post-task | 任务体验 | 主观难度、认知负荷、困惑点、注意力波动 | 验证 gaze/pointer/latency 是否对应真实体验 |
| Post-task | 支持需求 | 希望获得哪类支持、是否觉得某类支持有帮助 | 连接第一阶段状态线索与第二阶段支持需求 |
| Post-task | 可访问性 | 字体、行距、按钮、页面节奏、说明是否清楚 | 改进页面风格和 AOI 设计 |
| Post-task | 开放反馈 | 哪里不舒服、哪里误导、哪里希望系统不要判断 | 提前发现伦理和信任风险 |

边界：SEN/SEND 或 disability 信息只应作为可选、自愿披露项；第一阶段问卷重点是学习状态、支持偏好和技术接受度，不把问卷做成诊断筛查。

### 4.0.1 关键概念与记录口径需要讲清楚

新版 PPT 还需要单独解释 TUT、困惑 probe 和 AOI，否则导师会知道我们“要收状态线索”，但不清楚这些状态线索如何操作化。

| 概念 | 操作定义 | 建议记录方式 |
| --- | --- | --- |
| TUT | Task-Unrelated Thought，任务无关想法；例如阅读时注意离开文本、想到别的事情。 | 每段后自陈：yes/no + 可选 1-5 程度。 |
| 困惑 probe | 刚才是否有不理解、卡住、需要回看或不知道怎么答的时刻。 | 每段后自陈：none/slight/moderate/high + 可选困惑位置。 |
| AOI | Area of Interest，预先定义的页面区域，如文本、题目、选项、支持按钮。 | 系统自动保存区域矩形，并把 gaze/pointer 映射到 `aoi_id`。 |

环境上下文字段也不应全部做成 yes/no：

| 字段 | 建议格式 | 原因 |
| --- | --- | --- |
| `lighting_condition` | `normal` / `dim` / `backlit` / `uneven` + `good/medium/poor` 质量评分 | 光照不是二值变量；略暗、逆光、不均匀的影响不同。 |
| `head_movement` | `none` / `slight` / `frequent` | 轻微头动和频繁转头影响不同，不适合只记 yes/no。 |
| `wearing_glasses` | `yes` / `no` / `unknown`，可补充反光备注 | 是否戴眼镜适合二值记录，但反光需要附加说明。 |
| `researcher_help_event` | 时间戳 + 帮助类型 | 区分解释题意、提醒坐姿、技术处理等不同干预。 |

原则：能自动记录的自动记录；不能自动可靠判断的，用固定选项手动记录；自由文本只作为补充，不作为主要分析字段。

### 4.1 字段页 1：第一批，行为过程字段

建议标题：

> **第一批字段：学生在页面上的行为过程**

| 技术字段 | 中文含义 | 为什么收 | 例子 |
| --- | --- | --- | --- |
| `pointer_move` / `mouse_x` / `mouse_y` | 鼠标移动到哪里 | 判断是否用鼠标辅助阅读、是否在选项间犹豫 | 鼠标停在某个选项上但迟迟不点 |
| `click` / `pointer_down/up` | 点击了哪里、什么时候点 | 判断选择行为和页面互动 | 点击“下一题”前停顿很久 |
| `scroll_y` / `delta_y` | 页面滚动位置和方向 | 判断跳读、回看、停留位置 | 多次向上滚回文本说明在回看 |
| `response_latency_ms` | 题目出现到回答的时间 | 比只看答对/答错更能反映犹豫和负荷 | 答对但反应很慢，可能仍然吃力 |
| `answer_changed` | 是否改过答案 | 反映不确定性 | 先选 A，后改成 C |

### 4.2 字段页 2：第二批，眼动质量与页面位置

建议标题：

> **第二批字段：眼动数据是否可信、看的是哪里**

| 技术字段 | 中文含义 | 为什么收 | 例子 |
| --- | --- | --- | --- |
| `gaze_x` / `gaze_y` | 眼睛估计看的屏幕坐标 | 原始眼动数据 | 视线集中在题干还是选项 |
| `valid_gaze` | 当前眼动点是否有效 | 避免把坏数据当成行为差异 | 摄像头没抓到眼睛时标记无效 |
| `offscreen` | 视线是否跑出屏幕 | 判断分心或 tracking 失败 | 频繁 offscreen 可能是转头或校准差 |
| `aoi_id` / `aoi_type` | 当前区域属于文本、题目、选项还是按钮 | 把裸坐标翻译成“看了哪里” | 看选项区时间很长 |
| `calibration_error_px` | 校准误差大小 | 判断该轮眼动能不能信 | 校准误差太大则该轮降权或剔除 |

### 4.3 字段页 3：第三批，环境与远程协助上下文

建议标题：

> **第三批字段：解释数据差异所需的实验上下文**

| 技术字段 | 中文含义 | 为什么收 | 例子 |
| --- | --- | --- | --- |
| `lighting_condition` | 光照情况 | 解释追踪质量差异 | 正常光、略暗、逆光 |
| `wearing_glasses` | 是否戴眼镜 | 解释 P002 这类采样率下降 | 戴眼镜可能反光或遮挡眼睛 |
| `head_movement_instruction` | 是否要求尽量不转头 | 区分自然阅读和严格 baseline | P005 是严格不转头 baseline |
| `fatigue` | 主观疲劳程度 | 解释停顿、慢反应、低注意 | 受试者很累时更容易慢答 |
| `researcher_help_event` | 研究者是否提示或帮助 | 区分自然行为和被帮助后的行为 | Zoom 中研究者解释了题意 |
| `zoom_recording_id` | Zoom 录制编号 | 方便事后回看但不混入主 CSV | 只存编号，不在网页保存视频 |

### 4.4 字段页 4：支持需求与主观感受

建议标题：

> **第四批字段：学生觉得自己需要什么支持**

| 技术字段 | 中文含义 | 为什么收 | 例子 |
| --- | --- | --- | --- |
| `preferred_support` | 希望得到哪类支持 | 与 functional support needs 直接相关 | 需要摘要、提示、更多时间 |
| `support_option_used` | 是否使用了支持工具 | 区分“需要”和“实际使用” | 点击了术语解释 |
| `support_helpful_rating` | 支持是否有帮助 | 判断支持工具是否有效 | 摘要帮助理解文本 |
| `instruction_clarity` | 是否理解任务说明 | 避免误把任务困惑当学习困难 | 没看懂题目要求 |
| `tech_trust_rating` | 是否信任系统 | 对应 A4 的技术可信度问题 | 担心系统误判自己 |

## 5. 长期路线必须分段：不能一开始就跳到 SEN 识别或自适应系统

用户指出的这个问题是成立的，而且比单页 PPT 结构更重要。当前版本容易给导师造成一种印象：

> 我们想研究 SEN/SEND 学生，所以直接找 SEN/SEND 学生，采集眼动和平台行为，然后做自适应系统。

这个路线有三个现实风险：

1. **样本风险**: 如果一开始只找 SEN/SEND 学生，样本量可能很小，而且群体内部差异极大，模型很难稳定。
2. **伦理风险**: 如果表达成“识别 SEN 学生”或“分类 SEN 学生”，会触及标签化、歧视和误判风险。
3. **方法风险**: webcam gaze、鼠标、滚动和问卷只能提供学习过程线索，不能直接推出“这个学生是什么类型”或“需要什么支持”。

因此，路线应从“识别 SEN 身份”改成：

> 先验证普通笔记本和平台过程数据能否捕捉学习状态与支持需求线索；再在自愿披露和伦理边界下，探索这些线索是否能帮助 SEN/SEND 学生获得更合适的学习支持；最后才进入自适应系统设计。

### 5.1 第一阶段不应以“专门识别 SEN 学生”为目标

更稳妥的第一阶段目标是：

> 在普通学生样本或混合样本中，验证我们能否用 gaze、pointer、scroll、response latency、probe 和主观报告，区分不同的学习过程状态。

这里可以研究的状态包括：

| 状态/线索 | 可以尝试观察的信号 | 必须搭配的验证方式 |
| --- | --- | --- |
| 注意力集中/涣散 | gaze 有效率、offscreen、AOI 停留、scroll 回看、反应时间 | thought probe、任务表现、主观注意评分 |
| 困惑/认知负荷高 | 长停顿、反复回看、选项区犹豫、改答案 | difficulty rating、题目正确率、事后访谈 |
| 压力或不适 | 反应变慢、操作中断、主观疲劳/压力升高 | stress/fatigue rating，不能只靠眼动判断 |
| 需要支持 | 主动请求提示、偏好更多时间、需要摘要/术语解释 | support preference、support helpfulness rating |

这一步可以使用 classification 或 clustering，但口径要谨慎：

- **classification**: 只在有明确标签时使用，例如 probe 标记“走神/未走神”、主观压力高/低、是否请求支持。
- **clustering**: 可用于探索行为模式，例如“稳定阅读型”“频繁回看型”“低质量采集型”“高犹豫型”。
- 不应把这些模型说成“识别 SEN 学生”。更准确的说法是“识别学习过程状态或支持需求线索”。

### 5.2 SEN/SEND 应作为后续聚焦，而不是第一步的分类标签

SEN/SEND 在研究中仍然重要，但角色要换：

| 不建议的说法 | 更稳妥的说法 |
| --- | --- |
| 用眼动和平台数据识别 SEN 学生 | 在自愿披露/伦理批准下，探索 SEN/SEND 学生是否有不同的学习过程支持需求 |
| 给学生分类 | 识别可解释的学习状态和支持偏好 |
| 自动判断谁需要帮助 | 提供可选、可解释、可撤回的支持建议 |
| 系统根据身份调节 | 系统根据当前任务状态和用户选择调节 |

这样可以避免把研究做成有歧视风险的身份识别系统，同时保留 SEN/SEND 作为最终服务对象。

### 5.3 第一小段计划：先做可验证的学习状态 pilot

新版 PPT 应该增加一页：

> **第一阶段小计划：先验证学习状态线索，而不是直接做 SEN 分类**

建议第一小段目标：

1. 招募普通大学生或混合样本，不强制要求 SEN/SEND 身份。
2. 使用短文本阅读或学习材料任务。
3. 收集 gaze、鼠标、滚动、反应时间、AOI、校准质量。
4. 每段任务后收集 thought probe、difficulty、stress/fatigue、support preference。
5. 用简单模型或聚类探索：数据能否区分注意力状态、困惑程度、支持请求倾向。
6. 只报告群体层面模式，不做个人诊断或身份判断。

这个阶段的成功标准不是“识别 SEN”，而是：

| 成功标准 | 具体含义 |
| --- | --- |
| 数据可采 | 普通笔记本可稳定采集 gaze + interaction logs |
| 状态有信号 | 至少部分行为特征与 probe/主观评分/任务表现有关 |
| 解释可行 | AOI、回看、停顿、请求支持等能被非技术导师理解 |
| 风险可控 | 不保存网页视频，不做诊断，不做身份分类 |

### 5.4 自适应系统设计不能丢，但要放在第三阶段

一开始的三大块里确实还有“自适应系统设计”，这一块不能丢，但现在不宜过早做成完整系统。合理顺序应是：

| 阶段 | 研究问题 | 输出 |
| --- | --- | --- |
| 阶段 1：学习状态线索 | 数据能否反映注意、困惑、压力、支持请求等状态？ | 特征、标签、聚类、初步模型 |
| 阶段 2：支持需求映射 | 哪些状态对应哪些支持偏好？SEN/SEND 学生是否有特殊模式？ | 支持需求 taxonomy、用户访谈、共设计 |
| 阶段 3：自适应支持原型 | 系统如何根据状态和用户选择提供支持？ | 摘要、提示、延时、术语解释、任务分步等 adaptive features |
| 阶段 4：效果评估 | 自适应支持是否改善理解、负荷、体验和公平性？ | 对照实验、可解释性、伦理评估 |

建议 PPT 对导师的表述：

> 我们不会在第一阶段声称系统能识别 SEN 学生。第一阶段只验证学习状态线索；第二阶段才讨论这些线索如何与 SEN/SEND 支持需求相连；第三阶段再把这些线索转化为自适应支持功能。

## 7. 面向导师的口径调整

旧版语气：

> 我们已经完成了 WebGazer MVP，并计划下一版补字段。

新版应改为：

> 我们发现一个文献和方法之间的空白：已有研究支持 SEN/SEND 高等教育学生存在具体支持需求，也有研究用 WebGazer 或过程数据分析学习任务，但把普通笔记本眼动、平台交互和支持需求放在大学 SEN/SEND 场景中系统验证的工作还不充分。因此我们先做了一个 feasibility proof，证明采集链路可行；现在希望导师确认下一步任务场景、页面设计和数据边界。

## 8. 当前应该怎么改 80

## 9. 建议最后一页导师问题

1. 是否同意把研究定位为 **SEN/SEND / functional support needs 的学习过程支持**，而不是学生身份分类？
2. 下一版任务是否先采用 **短文本阅读理解 + thought probe + 支持偏好询问**？
